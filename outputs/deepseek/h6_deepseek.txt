As a QA engineer, I want to execute a comprehensive acceptance test suite against a fully deployed Spring XD cluster (admin + multiple containers) that validates end-to-end stream processing across all supported message transports (RabbitMQ, Kafka, Redis, and file-based transports) so that I can ensure consistent behavior regardless of the underlying messaging infrastructure.

As a QA engineer, I want the acceptance test suite to cover all out-of-the-box source, processor, and sink modules in various composition patterns (e.g., source→processor→sink, source→multiple sinks, fan-out/fan-in topologies) so that I can verify module interoperability and detect integration regressions before each release.

As a release manager, I want the acceptance test harness to include data validation checks (e.g., record counts, checksums, and schema conformity) at each pipeline stage, comparing expected vs. actual output so that I can certify that data integrity is maintained end-to-end across all module combinations.

As a QA engineer, I want the acceptance tests to include failure recovery scenarios such as container restarts, transport broker failures, network partitions, and module deployment failures, with automated verification that the cluster self-heals and resumes processing without data loss so that I can validate production-grade resilience.

As a developer, I want the acceptance test framework to produce a consolidated, machine-readable report (e.g., JUnit XML, JSON, or HTML) that breaks down results by transport type, module pair, and test scenario, with attached logs and metrics, so that I can quickly isolate root causes of any failures introduced by code changes.
